SECTION 2

解決する課題

このツールが解こうとしているのは
「解いた」で終わりにせず、
「なぜ間違えたか」まで記録する

学習ツールの多くは「正誤判定」で終わります。でも理解が定着しない本当の理由は、「なぜ間違えたか」が記録されないからです。3つを1画面に並べることで、解答から振り返りまでを一気に完結させます。

1 　解く
正解は合っていた？
OX・4択で解答。「解答する」で即座に正誤判定。
2 　振り返る
どのくらい理解できた？
理解度をA/B/Cで記録。次回の優先順位を自動で決める。
3 　分析する
なぜ間違えた？
知識不足・理解不足・ケアレスミス… 原因を6項目から選んでメモを残す。
3つの視点を 4ペインに割り当てる
それぞれを別のペインに置けば、視線が画面の中で完結する
PANE 1+2 → 解く 問題リスト → 解答で学習を進める
PANE 3 → 振り返る 理解度A/B/Cで次回の優先順位を決める
PANE 4 → 分析する 解説・原因6項目・メモで「なぜ？」を記録
○
3つを1つの画面で完結させることが、このツールの存在理由です。逆に言えば、「なぜ間違えたか」を記録したくない学習には、このツールは要りません。
✦
SECTION 3

工夫したポイント

特にこだわったのは
機能を増やすのではなく、
迷いを減らす方向に寄せ切ること

「学習の便利機能」を増やすほど、画面はごちゃごちゃして、肝心の振り返りが遅れます。今回は引き算の方向で4つの工夫を入れました。

工夫①
左 → 右の一方向で情報が細かくなる

4つのペインは「ざっくり → くわしく」の順で並んでいて、逆方向の情報は置きません。たとえばPane 4（原因分析）の中に「問題を切り替えるリスト」を入れたくなりますが、入れていません。

›
PANE 2
問題と解答
›

›
PANE 4

情報の細かさは「左 → 右」だけに流す
✓ 効果: 視線が迷子にならない。「いま自分は問題を解いているのか、振り返っているのか」が、画面のどこを見ているかで自然に分かります。
A
工夫②
理解度バッジで次回の優先問題が一目で分かる

問題リスト（Pane 1）の各行には、理解度バッジ（A/B/C）を表示しています。C（不安）→ B（だいたい）→ A（完全）の優先順でソートされるため、次に解くべき問題が一目で分かります。

× よくあるパターン
問題番号順に並ぶだけ。どれを優先すればいいか毎回考える必要がある。
✓ このツール
Cバッジ（不安）が上に来る。リストを見るだけで「今日はこれをやろう」が決まる。
✓ 効果: 「どれを解こうか」と迷う1ステップが消える。学習を始めるハードルが下がります。
→|
工夫③
解答後にペインが順番に解放される

Pane 3（理解度）は解答後に解放。Pane 4（原因分析）は理解度を選んだ後に解放されます。一度に全部見せない設計により、学習フローが自然に一方向に流れます。

問題選択 → 解答 → 理解度A/B/C → 解説・原因・メモ
✓ 効果: 「何を先にやればいいか」で迷わない。フローに沿って動くだけで振り返りが完了します。
□
工夫④
原因を6項目から選ぶだけで記録できる

「なぜ間違えた？」を毎回文章で書くのは続きません。知識不足・理解不足・読み飛ばし・勘違い・ケアレスミス・時間不足の6項目からチェックするだけで記録が完了します。

✓ 効果: 「書く手間」がゼロになり、振り返りの継続率が上がる。積み上がった記録が「自分はケアレスミスが多い」などのパターン発見につながります。
4つの工夫に共通すること
機能を足すのではなく、「迷いどころ」を1つずつ消す

優先ソート・段階的解放・チェックリスト。どれも「ユーザーがどこで迷うか」を観察して、その芽を1つずつ摘んでいく作業でした。便利機能を1つ足すより、迷いを1つ消すほうが、画面はずっと使いやすくなります。

SECTION 4

苦戦したポイント

苦戦のいちばんのポイント ─ 「4ペインでなければいけない理由」
コードよりも、「何を作らないか」を
決めるところ

絵を描いたりコードを書いたりする前に、考えるだけで何度も詰まったのがここです。特に「なぜ4ペインなのか」——次の3つの分岐で大いに迷いました。

苦戦①
何をテーマにするか — 学習に絞るか、汎用にするか

最初は「UIデザイン / プログラミング / ビジネス知識の3分野を1つに同梱」や「どの科目でも使える汎用テンプレート」も考えました。たくさんカバーした方が「便利そう」に見えるからです。

✓
結論: 分野を複数並べると、どの学習にも中途半端なお手本にしかなりません。ニールセン10原則1分野に絞り切って、その完成度を上げる方が、見る人が「自分の学習にも使える」と目標を持てると判断しました。
苦戦②
4ペインの役割をどう分けるか

4ペインにすると決めても、どの情報をどこに置くかが難題でした。最初は「問題／解答／解説／メモ」のような構成も考えました。しかし実際に並べると、「今どこを見ればいいのか分からない」「解説を読む前に答えが見えてしまう」という問題が発生しました。

案A
問題→解答→解説→メモ
→ 解答が先に見えてしまう
案B
問題→メモ→解答→解説
→ 学習の流れが分断される
案C ✓（採用）
問題一覧→問題→原因分析→解説
→ 左から右へ自然に深くなる
✓
結論: 4ペインは単に画面を4分割することではありません。「ざっくり理解→詳しく理解」という思考の流れを、左から右へ視覚化するための構造だと気づきました。
苦戦③
「原因分析」を入れるべきか

最初は「問題を解く→答えを見る」だけのシンプルな構成も考えました。その方が画面もスッキリ見えるからです。しかし実際に使うと、「なぜ間違えたのか」「なぜ正解できたのか」が残らず、学習が一回で終わってしまいました。

原因分析なし
問題→解答→終了
原因分析あり ✓
問題→自信度選択→原因分析→解答
✓
結論: 今回のツールの目的は「正解すること」ではなく「理解を深めること」です。そのためには原因分析ペインが必要でした。4ペインになった理由も、実はこの原因分析を独立した場所として確保したかったからです。
3つの苦戦から学んだこと
「誰でも使えるツール」を狙うと、
結局は誰にも噛み合わない画面になる

みんなが使えるように作ろうとするほど、結局はどの学習にも噛み合いません。自分の中心の課題を1つに絞る勇気と、絞るために考え抜く根気。この2つが、画面を作る前に必要でした。コードを書く時間より、決めきれずに立ち止まる時間のほうが長かったのが正直なところです。
